Changelog Template

Purpose
Provide a consistent narrative structure for release notes enhancing clarity for users and contributors.

Version Header
Version: X.Y.Z  
Release Date: YYYY‑MM‑DD  
Status: Stable / Beta / Patch

Summary
Concise overview highlighting primary value delivered (one or two sentences).

Added
• New features, modules, or capabilities introduced in this release.
• Example: Introduced TimeEntry aggregation on invoice overview.

Changed
• Modifications to existing behavior, UI adjustments, performance improvements.
• Example: Refined task detail layout for clearer deadline visibility.

Fixed
• Bugs resolved, regressions addressed, stability improvements.
• Example: Corrected case reference persistence in expense records.

Removed
• Deprecated features eliminated, obsolete dependencies dropped.
• Example: Removed legacy placeholder notification stubs.

Security
• Notable security hardening measures, vulnerability patches, dependency upgrades addressing CVEs.
• Example: Added input validation for invoice name fields.

Migration Notes
• Required actions for users upgrading (e.g., data migrations, configuration changes).  
• Example: Open application once to trigger Hive box schema upgrade.

Known Issues
• Outstanding defects or limitations acknowledged but unresolved.  
• Example: Ads may intermittently fail on specific device models.

Deprecations
• Features marked for removal in future versions with recommended alternatives.  
• Example: Legacy PDF renderer scheduled for replacement by optimized pipeline.

Acknowledgements
• Recognize significant contributor efforts.  
• Example: Thanks to contributor for implementing calendar view enhancements.

Next Focus
• High‑level preview of planned roadmap initiatives post‑release.

Usage Guidance
Maintain chronological order from newest to oldest releases. Ensure each entry is self‑contained and avoids ambiguous terminology.

Summary
Structured release documentation improves transparency, upgrade confidence, and historical traceability.
